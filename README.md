# 社区慢病随访复核看板

本地社区卫生服务中心慢病随访数据质量核查与异常复核管理系统。

## 功能特性

- 📥 **三类数据导入**: 居民名册、预约计划、随访记录 CSV 文件导入
- 🔍 **导入前数据质量预检**: 上传后先给出预检结果（按严重程度分组，显示行号/字段/修复建议），再决定是否正式导入
- 🛡️ **预检规则覆盖**: 缺失必填列/值、日期格式异常、数值字段非法、居民编号跨表找不到、同一居民同日重复随访
- ⚙️ **预检可配置项**: 最多展示错误数、是否允许警告继续导入，刷新页面后持久化保留
- 🚫 **导入保护机制**: 遇到重复文件或预检未通过时不覆盖旧数据，提供明确提示和完整操作日志
- ✅ **灵活导入策略**: 允许只导入通过预检的数据，也允许取消
- 📤 **预检报告导出**: 支持 CSV/JSON 格式，导出内容与当前筛选条件严格一致
- 🔍 **五类异常自动识别**:
  - 逾期未访（预约已过但无随访）
  - 未预约到访（有随访但无对应预约）
  - 重复随访（同一居民同一天多条随访）
  - 指标越界（血压/血糖超出正常范围）
  - 居民不在名册（记录的居民编号不存在）
- 🎛️ **多维筛选**: 按站点、护士、异常类型、复核状态筛选
- ✅ **异常复核**: 标记待处理、已确认、忽略、需上门四种状态，支持备注和处理人
- 🛡️ **质控规则配置**: 可配置逾期阈值、血压血糖上下限、未预约匹配窗口等规则，支持版本管理
- ⏮️ **规则回放**: 规则版本历史管理，预览差异后切换版本，保护人工复核结果
- 📒 **操作日志**: 记录导入、规则变更、批量重算、导出、复核状态修改，支持筛选导出
- 💾 **本地持久化**: 所有数据（含复核状态、规则、日志）存储在 localStorage，刷新重启不丢失
- 📤 **数据导出**: JSON/CSV 格式导出当前筛选结果

## 功能模块

### 三大模块导航

系统通过顶部 Tab 切换三大模块：
- **异常看板**: 数据导入、异常列表展示、筛选与复核
- **质控规则**: 规则配置、版本管理、批量重算
- **操作日志**: 操作记录查询、筛选与导出

## 质控规则配置

### 可配置规则项

| 规则项 | 说明 | 默认值 |
|--------|------|--------|
| 规则启用开关 | 关闭后仅识别"重复随访"和"居民不在名册" | 启用 |
| 逾期未访阈值 | 超过多少天未随访算逾期 | 0天 |
| 未预约匹配窗口 | 随访前后多少天内有预约视为匹配 | 3天 |
| 收缩压上限/下限 | 收缩压正常范围 | 90-140 mmHg |
| 舒张压上限/下限 | 舒张压正常范围 | 60-90 mmHg |
| 血糖上限/下限 | 血糖正常范围 | 3.9-7.0 mmol/L |
| 需要上门的状态映射 | 哪些异常类型在首次识别时，复核状态自动标记为"需上门"而非默认的"待处理" | 逾期未访、指标越界 |

### 操作流程

1. **修改规则**: 在"质控规则" Tab 调整各项阈值
2. **预览差异**: 点击"预览差异"查看规则变更对现有异常的影响
3. **保存版本**: 填写版本名称后点击"保存并应用"，自动生成新版本
4. **恢复默认**: 点击"恢复默认"一键回到出厂规则

### 四、人工复核结果保护（冲突处理）

**验收目标**: 已确认/忽略的异常不会被规则变更或重算静默覆盖

**复现步骤**:
1. 导入样例数据，在异常列表中选择 2-3 条异常
2. 将其中一条状态改为"已确认"，另一条改为"忽略"，第三条保留"待处理"
3. 切换到"质控规则" Tab，大幅度修改规则（如将血压上下限设为极端值）
4. 点击"保存并应用"
5. 回到"异常看板"，找到刚才操作的三条记录
6. 验证：
   - 状态为"已确认"和"忽略"的记录，状态和备注完好无损
   - 即使按新规则该记录不应被识别为异常，它依然保留在列表中（受保护）
   - 状态为"待处理"的记录，按新规则正常更新
7. 点击顶部"重算"按钮，选择"保护状态并重算"
8. 再次验证：已确认/忽略的记录仍然存在且状态未变

## 数据质量预检（新增功能）

### 预检规则覆盖

| 规则 | 严重程度 | 说明 |
|------|----------|------|
| 缺失必填列 | error | CSV 表头缺少居民编号、姓名等必填字段 |
| 缺失必填值 | error | 某一行缺少必填字段内容 |
| 日期格式异常 | error | 预约日期/随访日期无法解析 |
| 数值字段非法 | error | 收缩压/舒张压/血糖不是数字 |
| 居民编号跨表找不到 | warning | 预约或随访引用的居民编号不在居民名册 |
| 同一居民同日重复随访 | warning | 同一居民编号同一天有多条随访 |

### 场景 A：预检通过 — 直接导入

1. 重置所有数据
2. 导入正常的 `residents.csv`
3. 观察：预检对话框弹出，显示"预检通过，无错误和警告"
4. 点击"确认导入"
5. 验证：
   - 导入成功，居民数量显示为 10
   - "操作日志" Tab 中出现"预检通过导入"日志

### 场景 B：警告继续导入 vs 只导入有效数据（validOnly 跳过 warning 行）

1. 先导入正常的 `residents.csv`（已有 R001-R010）
2. 准备一个随访 CSV，包含以下行：
   - R001,2026-06-01（合法居民，正常数据）
   - R999,2026-06-02（不存在的居民，触发"居民编号不存在" warning）
   - R001,2026-06-01（同日重复，触发"同一居民同日重复随访" warning）
3. 导入该随访 CSV
4. 观察：预检对话框显示 3 条 warning（R999 不存在 + 2 条 R001 同日重复），无 error
5. 验证：
   - 对话框底部出现"警告继续导入"和"仅导入有效数据"两个按钮
   - 点击"仅导入有效数据"：只导入 1 条 R001 的非重复合法记录，R999 和同日重复的 R001 行全部被跳过，导入成功提示显示"跳过 N 行无效/警告数据"
   - 异常统计中不会出现 R999 相关的"居民不在名册"异常
   - 或点击"警告继续导入"：全部 3 条导入（按 allowWarningContinue 配置），后续异常统计会出现 R999 和重复随访警告

> **核心区别**："仅导入有效数据"同时跳过 error 行 **和** warning 行（居民不存在、同日重复随访等）；"警告继续导入"按配置决定是否允许 warning 行入库。

### 场景 C：预检失败拦截 — 不覆盖旧数据

## 数据导入批次追踪与撤销（新增功能）

**验收目标**：每次居民、预约、随访 CSV 真正入库后自动生成导入批次，支持跨刷新持久化、一键撤销最近批次、人工复核保护、冲突拦截、完整导出与操作日志。

### 场景 A：批次自动生成 + 跨刷新持久化

1. 依次导入 `residents.csv` → `appointments.csv` → `followups.csv`（每次导入走预检→确认入库流程）
2. 切到 **"导入批次"** Tab（位于"质控规则"和"操作日志"之间）
3. 验证：
   - 表格显示 3 条记录，按导入时间倒序（随访在最上）
   - 每条包含：数据类型、文件名、导入条数、跳过条数、预检摘要徽标、操作者、导入时间
   - 展开任意行可见：批次 ID、文件哈希、导入模式（all/validOnly/direct）、预检统计、快照数据量
4. **刷新浏览器**（F5）或关闭后重新打开
5. 验证："导入批次" Tab 中 3 条记录依然存在，顺序、字段、展开详情均完好
6. 同样操作验证"操作日志" Tab 中 3 条 "生成导入批次" 日志也持久化保留

### 场景 B：撤销最近一次成功导入 — 数据同步还原

1. 完成场景 A 后，当前有 3 条批次：随访（最新）、预约、居民
2. 点击顶部 **"撤销最近批次"** 按钮（或点击表格中随访行"撤销"按钮）
3. 弹出确认面板，说明将还原随访数据、异常统计、复核状态，同时保留 CONFIRMED/IGNORED 人工复核
4. 点击确认撤销
5. 验证：
   - 顶部绿色提示"撤销成功"，显示还原的随访条数和受保护异常数
   - 随访批次行打上 **"已撤销"** 徽标，显示撤销时间和撤销人
   - 回到"异常看板"，随访相关异常（如逾期未访、重复随访、指标越界）恢复到导入随访前的数量
   - `npm run test:qc` 批次测试7 验证三类数据和 importedFileHashes 同步还原

### 场景 C：人工复核保护 — 撤销不误删已确认/忽略的复核结果

1. 导入 `followups.csv`，在异常看板中挑 2 条异常：1 条"确认"（CONFIRMED，写备注）、1 条"忽略"（IGNORED，写处理人）
2. 记录这 2 条异常的 anomalyId 和处理内容
3. 再导入一份新的随访数据（产生第二个随访批次）
4. 现在**撤销第二个随访批次**
5. 验证：
   - 之前的 2 条人工复核结果（CONFIRMED + IGNORED）依然保留在异常列表中
   - 它们的状态、备注、处理人都未被撤销操作覆盖
   - 其余未复核的异常按快照还原
   - `npm run test:qc` 批次测试2 验证快照内 + 快照外 CONFIRMED/IGNORED 全部受保护

### 场景 D：冲突拦截 — 四类场景给出明确提示

**D1. NOT_LATEST（非最新批次不可直接撤销）**
1. 有 3 条批次：随访、预约、居民（倒序）
2. 点击"居民"批次行的"撤销"按钮
3. 验证：红色提示 **"仅允许撤销最新未撤销批次，请先撤销 followups.csv（2025-07-03...）"**
4. `npm run test:qc` 批次测试3 覆盖

**D2. DEPENDENCY_EXISTS（居民名册被下游依赖）**
1. 先撤销随访批次（解除随访对居民的依赖）
2. 再尝试撤销居民批次，但此时还有预约批次未撤销
3. 验证：红色提示 **"存在 1 个下游批次依赖此居民名册，需先撤销它们：appointments.csv"**
4. `npm run test:qc` 批次测试4 覆盖

**D3. HASH_CONFLICT（文件哈希不一致）**
1. 对同一批数据，通过控制台修改 localStorage 中该批次的 fileHash 字段为其他值
2. 尝试撤销该批次
3. 验证：红色提示 **"当前文件哈希与批次记录不一致，可能数据已被其他操作修改，撤销已阻断"**
4. `npm run test:qc` 批次测试5 覆盖

**D4. ALREADY_REVERTED（不可重复撤销）**
1. 撤销一个批次成功后，该行"撤销"按钮变为禁用的"已撤销"
2. 通过控制台强制再次调用 `revertBatch(batchId)`
3. 验证：返回 blockedReason=ALREADY_REVERTED，revertedAt 和 revertedBy 字段已记录
4. `npm run test:qc` 批次测试6 覆盖

### 场景 E：批次导出 — JSON/CSV 字段完整
