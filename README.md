# 社区慢病随访复核看板

本地社区卫生服务中心慢病随访数据质量核查与异常复核管理系统。

## 功能特性

- 📥 **三类数据导入**: 居民名册、预约计划、随访记录 CSV 文件导入
- 🔍 **导入前数据质量预检**: 上传后先给出预检结果（按严重程度分组，显示行号/字段/修复建议），再决定是否正式导入
- 🛡️ **预检规则覆盖**: 缺失必填列/值、日期格式异常、数值字段非法、居民编号跨表找不到、同一居民同日重复随访
- ⚙️ **预检可配置项**: 最多展示错误数、是否允许警告继续导入，刷新页面后持久化保留
- 🚫 **导入保护机制**: 遇到重复文件或预检未通过时不覆盖旧数据，提供明确提示和完整操作日志
- ✅ **灵活导入策略**: 允许只导入通过预检的数据，也允许取消
- 📤 **预检报告导出**: 支持 CSV/JSON 格式，导出内容与当前筛选条件严格一致
- 🔍 **五类异常自动识别**:
  - 逾期未访（预约已过但无随访）
  - 未预约到访（有随访但无对应预约）
  - 重复随访（同一居民同一天多条随访）
  - 指标越界（血压/血糖超出正常范围）
  - 居民不在名册（记录的居民编号不存在）
- 🎛️ **多维筛选**: 按站点、护士、异常类型、复核状态筛选
- ✅ **异常复核**: 标记待处理、已确认、忽略、需上门四种状态，支持备注和处理人
- 🛡️ **质控规则配置**: 可配置逾期阈值、血压血糖上下限、未预约匹配窗口等规则，支持版本管理
- ⏮️ **规则回放**: 规则版本历史管理，预览差异后切换版本，保护人工复核结果
- 🧪 **方案沙盒**: 护士长可把规则另存为草稿，用现有数据预跑看差异，确认后再应用；支持 JSON 导入导出；可撤销最近应用并保留人工复核结果
- 📒 **操作日志**: 记录导入、规则变更、批量重算、导出、复核状态修改，支持筛选导出
- 💾 **本地持久化**: 所有数据（含复核状态、规则、日志）存储在 localStorage，刷新重启不丢失
- 📤 **数据导出**: JSON/CSV 格式导出当前筛选结果

## 功能模块

### 三大模块导航

系统通过顶部 Tab 切换三大模块：
- **异常看板**: 数据导入、异常列表展示、筛选与复核
- **质控规则**: 规则配置、版本管理、批量重算
- **操作日志**: 操作记录查询、筛选与导出

## 质控规则配置

### 可配置规则项

| 规则项 | 说明 | 默认值 |
|--------|------|--------|
| 规则启用开关 | 关闭后仅识别"重复随访"和"居民不在名册" | 启用 |
| 逾期未访阈值 | 超过多少天未随访算逾期 | 0天 |
| 未预约匹配窗口 | 随访前后多少天内有预约视为匹配 | 3天 |
| 收缩压上限/下限 | 收缩压正常范围 | 90-140 mmHg |
| 舒张压上限/下限 | 舒张压正常范围 | 60-90 mmHg |
| 血糖上限/下限 | 血糖正常范围 | 3.9-7.0 mmol/L |
| 需要上门的状态映射 | 哪些异常类型在首次识别时，复核状态自动标记为"需上门"而非默认的"待处理" | 逾期未访、指标越界 |

### 操作流程

1. **修改规则**: 在"质控规则" Tab 调整各项阈值
2. **预览差异**: 点击"预览差异"查看规则变更对现有异常的影响
3. **保存版本**: 填写版本名称后点击"保存并应用"，自动生成新版本
4. **恢复默认**: 点击"恢复默认"一键回到出厂规则

### 二、质控规则配置与生效

**验收目标**: 修改规则后异常识别立即按新规则计算

**复现步骤**:
1. 导入样例数据后，切换到"质控规则" Tab
2. 记录当前"指标越界"类异常数量

8. 将"血糖上限"从 7.0 改为 11.0（放宽标准）
9. 不预览，直接"保存并应用"，填写版本名"放宽血糖标准"

### 三、规则版本管理与回放

**验收目标**: 版本可追溯、可切换，切换时有差异预览

### 四、人工复核结果保护（冲突处理）

**验收目标**: 已确认/忽略的异常不会被规则变更或重算静默覆盖

**复现步骤**:
1. 导入样例数据，在异常列表中选择 2-3 条异常
2. 将其中一条状态改为"已确认"，另一条改为"忽略"，第三条保留"待处理"
3. 切换到"质控规则" Tab，大幅度修改规则（如将血压上下限设为极端值）
4. 点击"保存并应用"
5. 回到"异常看板"，找到刚才操作的三条记录
6. 验证：
   - 状态为"已确认"和"忽略"的记录，状态和备注完好无损
   - 即使按新规则该记录不应被识别为异常，它依然保留在列表中（受保护）
   - 状态为"待处理"的记录，按新规则正常更新
7. 点击顶部"重算"按钮，选择"保护状态并重算"
8. 再次验证：已确认/忽略的记录仍然存在且状态未变

## 数据质量预检（新增功能）

### 预检规则覆盖

| 规则 | 严重程度 | 说明 |
|------|----------|------|
| 缺失必填列 | error | CSV 表头缺少居民编号、姓名等必填字段 |
| 缺失必填值 | error | 某一行缺少必填字段内容 |
| 日期格式异常 | error | 预约日期/随访日期无法解析 |
| 数值字段非法 | error | 收缩压/舒张压/血糖不是数字 |
| 居民编号跨表找不到 | warning | 预约或随访引用的居民编号不在居民名册 |
| 同一居民同日重复随访 | warning | 同一居民编号同一天有多条随访 |

## 方案沙盒（新增功能）

**前置**: 先用场景 E 导出一份包含待办项的 JSON，再复制出多份副本做不同修改。

**复现步骤 — 冲突1：INVALID_JSON**:
1. 班次待办 → 导入 JSON Tab，上传一份内容损坏的 JSON 文件（如手工把花括号改错）
2. 验证：结果区显示红色提示，错误码 `[INVALID_JSON]`，并附"请检查文件是否为有效 JSON"建议

**复现步骤 — 冲突2：VERSION_MISMATCH**:
1. 手工编辑导出的 JSON，把顶层 `schemaVersion` 改成 `"0.0.1"`
2. 导入该文件，验证：结果区红色提示，错误码 `[VERSION_MISMATCH]`，错误信息明确列出"期望 X.X.X，实际 0.0.1"

**复现步骤 — 冲突3：DUPLICATE_NAME**:
1. 在本地先创建一个名为"今日交班"的清单
2. 再导入一份同样叫"今日交班"的 JSON
3. 验证：结果区红色提示，错误码 `[DUPLICATE_NAME]`，并说明"已存在同名清单请改名后再导入"

**复现步骤 — 冲突4：ANOMALY_NOT_FOUND / ANOMALY_UPDATED**:
1. 导出一份包含 N 条待办的 JSON
2. 手工把其中 1 条的 `anomalyId` 改成一个不存在的 ID（如 `NONEXIST001`）
3. 把另外 1 条的 `anomalyUpdatedAt` 改成比本地记录**更早**的时间
4. 导入该文件
5. 验证：
   - `[ANOMALY_NOT_FOUND]` 黄色 warning 1 条，明确提示对应居民 / anomalyId
   - `[ANOMALY_UPDATED]` 黄色 warning 1 条，提示"本地异常已更新，导入的是旧版本"
   - 汇总计数 `notFoundCount=1`、`updatedCount=1`
